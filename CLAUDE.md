# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this app is

mcClippy is a local-first macOS menu-bar clipboard-history app (Win+V style). Captured items are sealed with ChaChaPoly using a Keychain-resident key. Sensitive content is detected and stored with a placeholder preview so the real text never sits in plaintext anywhere on disk; it's only re-derived from the decrypted blob on explicit reveal at display time.

## Repo layout — note the doubled path

The git repo root is `mcClippy/mcClippy/` (yes, two levels). Swift sources live in `mcClippy/mcClippy/mcClippy/`. Build commands always reference the project file at `mcClippy/mcClippy/mcClippy.xcodeproj`. The Xcode project uses **synchronized file groups** (`PBXFileSystemSynchronizedRootGroup`), so new `.swift` files placed under `mcClippy/mcClippy/mcClippy/` are picked up by the build automatically — no `project.pbxproj` edits needed.

## Build, run, test

```bash
# Build (Debug, .accessory menu-bar app — no Dock icon)
xcodebuild -project mcClippy.xcodeproj -scheme mcClippy -configuration Debug -destination 'platform=macOS' build

# Build (Release, ad-hoc signed — same flow CI uses)
xcodebuild -project mcClippy.xcodeproj -scheme mcClippy -configuration Release -destination 'platform=macOS' \
  CODE_SIGN_IDENTITY="-" CODE_SIGN_STYLE=Manual DEVELOPMENT_TEAM="" \
  CODE_SIGNING_REQUIRED=YES CODE_SIGNING_ALLOWED=YES build

# Run unit tests — MUST be serial. Parallel mode has a pre-existing instability
# in the test bundle (the encryption round-trip Keychain access is the suspect).
xcodebuild -project mcClippy.xcodeproj -scheme mcClippy -destination 'platform=macOS' \
  -only-testing:mcClippyTests -parallel-testing-enabled NO test

# Single test
xcodebuild ... -only-testing:mcClippyTests/mcClippyTests/encryptionRoundTripsData -parallel-testing-enabled NO test
```

If `xcodebuild` errors with "tool requires Xcode but active developer dir is CommandLineTools", prefix with `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer xcrun`.

SourceKit's index frequently shows stale "Cannot find type 'Item' in scope" diagnostics for cross-file references. The `xcodebuild` output is the source of truth — ignore the SourceKit noise unless it persists after a full build.

## Release flow

Tag-driven via GitHub Actions (`.github/workflows/release.yml`):

```bash
git tag v1.1.x
git push origin v1.1.x
```

The workflow builds Release, ad-hoc signs, packages a zip + sha256, and publishes to GitHub Releases. The yml has commented-out stubs for switching to Developer ID + `notarytool` when an Apple Developer cert is available. `PrivacyInfo.xcprivacy` is bundled into `Contents/Resources/` automatically via the synchronized file group.

## Architecture — the parts that span files

The app is intentionally headless (`NSApp.setActivationPolicy(.accessory)` in `AppDelegate`). There is **no** `WindowGroup`. Two surfaces only:

1. `MenuBarExtra` — the menu-bar dropdown (`MenuBarControlsView` in `mcClippyApp.swift`).
2. `Settings` scene — `SettingsView` with three tabs (General / Exclusions / About).

The history panel is **not** a SwiftUI scene. It's a borderless `NSPanel` (`KeyablePanel`) owned by `PasteHistoryPanelController.shared` and hosting `ContentView` via `NSHostingView`. The controller is the single point of truth for show/hide, and it captures the previously frontmost app **before** activating mcClippy so auto-paste can return focus correctly.

### Keyboard routing — the non-obvious bit

`.onKeyPress` modifiers on the SwiftUI view tree **do not fire** while the search `TextField` is focused (which it always is on panel open). Instead, `PasteHistoryPanelController.show()` installs an `NSEvent.addLocalMonitorForEvents(matching: .keyDown)` that intercepts arrow / Return / Esc keys **before** they reach the TextField and posts notifications:

- `.panelMoveSelectionUp` / `.panelMoveSelectionDown`
- `.panelPasteSelected`
- `.panelDismiss`

`ContentView` observes these via `.onReceive`. The monitor is removed in `close()`. Do not move the key handling back to `.onKeyPress` — it will silently break navigation again.

### Capture and storage pipeline

Defined in `mcClippyApp.swift`:

1. `PasteboardMonitor.tick()` (Timer @ 0.7s) reads `NSPasteboard.general.changeCount`.
2. Skips if paused, in private mode, the changeCount is unchanged, or the frontmost app's bundle ID is in `AppExclusionStore`.
3. `PasteboardSerializer.snapshot(from:)`:
   - Rejects pasteboards advertising `org.nspasteboard.ConcealedType` / `TransientType` / `AutoGeneratedType`, the legacy 1Password type, or `com.apple.is-remote-clipboard`.
   - Computes `sourceIsPM` from `PasteboardSerializer.passwordManagerBundleIDs` — copies originating from any known password manager are force-flagged sensitive even if content doesn't match patterns. **Always add new password managers here, not via content heuristics.**
   - Falls back to `SensitiveContentDetector.looksSensitive` which checks keyword markers, Stripe/OpenAI `sk_`/`pk_` keys, GitHub PATs (`gh[pousr]_…`), JWTs (`eyJ.eyJ.…`), padded base64, and `looksLikePassword` (8–64 chars, no whitespace, no URL signature, ≥3 character classes).
4. `PasteboardMonitor.capture()` seals the raw blob via `EncryptionService.seal` and inserts an `Item` with `isEncrypted = true` and `plainTextPreview = sensitivePlaceholder` if sensitive.

### Decryption is centralized

Anything that needs the original bytes — restore to clipboard, image thumbnails, sensitive-reveal preview — must go through `PasteboardSerializer.decodedData(for:)` or `PasteboardSerializer.displayPreview(for:revealingSensitive:)`. Don't read `item.dataBlob` directly.

### SwiftData model migration

New `Item` properties (`isEncrypted`, `ocrText`, `ocrCompletedAt`) are all added with default values so SwiftData's lightweight migration handles old stores. `PasteboardMonitor.start()` runs `sanitizeSensitivePreviews` once per launch to scrub any legacy plaintext previews on items flagged sensitive. Don't remove that sweep.

### OCR

`OCRService.shared` is an actor that wraps `VNRecognizeTextRequest`. Triggered on demand from `ContentView.pasteAsText(_:)` when the user picks "Paste as Text" on an image row. Concurrency capped at 2 via `ConcurrencyPermitter`. Dedups by `Item.id` so the same image isn't extracted twice. Decodes through `PasteboardSerializer.decodedData(for:)` (same centralized path as everything else), downsamples to 4096 px before Vision, and writes the result back to the `Item` on `@MainActor`. After write, the OCR text is run through `SensitiveContentDetector.looksSensitive` and flips `isSensitive` if it hits. **Don't add an eager capture-time OCR path** — it was rejected for battery reasons.

### Shortcut handling

Global hotkey is a Carbon `EventHotKey` (not SwiftUI `KeyboardShortcut`). `GlobalShortcutManager.reload()` unregisters and re-registers whenever `ShortcutStore.current` changes. The manager also re-registers on `NSWorkspace.didWakeNotification` and `NSWorkspace.sessionDidBecomeActiveNotification` — Carbon hot keys silently stop dispatching after long sleep/wake cycles or `loginwindow` re-spawns, and re-arming via these is the fix. `ShortcutSpec` is `Codable` and persisted in `UserDefaults`. `ShortcutRecorderView` uses a local `NSEvent` monitor with both an Esc-cancel and a `didResignKey` escape hatch — if you change the recorder, keep both.

### Auto-paste

`AutoPaster.paste(into:)` activates the previously frontmost app, polls for `target.isActive` (up to ~400 ms / 8 × 50 ms retries) before posting a `CGEvent` for `⌘V` through `.cghidEventTap`. The HID tap is more reliable than the annotated session tap across Electron/Chromium destinations. Requires Accessibility trust (`AXIsProcessTrusted`).

The current build sets `ENABLE_APP_SANDBOX = NO` for direct distribution because there's no need for the sandbox-specific TCC dance when we're not going through the Mac App Store. A sandboxed MAS build *can* post `CGEvent`s — Apple exposes a dedicated `kTCCServicePostEvent` privilege (separate from full Accessibility) that permits `CGEvent.post(...)` under sandbox. This is what Maccy, Paste, Pastebot, and Flycut all do on the App Store. So the planned v1.2.0 MAS target keeps the same `AutoPaster` code; only the entitlements file differs.

## Conventions worth keeping

- `MainActor.assumeIsolated` is used to bridge from non-isolated callbacks (Carbon event handler, NotificationCenter observers) into MainActor state. Don't replace with `Task { @MainActor in … }` where the receiving code needs to act synchronously inside a hot path.
- `try? context.save()` is called explicitly after SwiftData inserts/deletes in `PasteboardMonitor` — autosave isn't relied on.
- Sensitive-content false positives are preferred over false negatives. Bias the heuristic toward flagging when in doubt; the user can reveal.
